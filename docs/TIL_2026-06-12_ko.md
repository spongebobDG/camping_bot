# TIL 2026-06-12 - Camping Bot ROS2 Bringup 정리

## 목표

캠핑장을 돌아다니며 위험을 감지하고, 사용자 요청에 따라 배송, 길안내, 대피, 경고 부저 같은 기능으로 확장할 수 있는 캠핑 로봇의 ROS2 기반 시스템을 구축했다.

이번 작업의 핵심은 하드웨어가 실제로 움직이는 상태에서 다음 흐름을 만드는 것이었다.

```text
센서 수신 -> ROS2 토픽 변환 -> RViz 확인 -> 주행 명령 -> 위험 감지 -> 미션 상태 통합 -> 순찰/부저/Nav2 확장
```

## 하드웨어 구성

- 배터리: 18650 3S2P
- 모터 드라이버: L298N
- 모터: JGB37 DC 모터
- 구동 방식: 4륜 구동
- 조향 방식: 앞바퀴 Ackermann 조향
- 모터/IMU 제어 보드: ESP32, IP `192.168.0.10`
- 라이다 제어 보드: ESP32-S3, IP `192.168.0.9`
- 카메라 보드: Seeed XIAO ESP32-S3 Sense, IP `192.168.0.11`
- 라이다: LDS14 TRANS V1
- IMU: MPU6050
- 부저: MM-FMD low-level-trigger 3핀 부저, ESP32 `GPIO13`

## 라이다 Bringup

ESP32-S3에서 LDS14 라이다 데이터를 UDP로 보내고, ROS2 노드에서 `LaserScan`으로 변환했다.

생성한 주요 토픽:

```text
/scan
/scan_fast
/scan_viz
```

확인한 점:

- 라이다 UDP 패킷이 정상 수신됨
- `/scan` 주파수는 약 15Hz 수준으로 확인됨
- RViz에서 QoS 문제 때문에 라이다가 안 보일 수 있었고, best-effort 설정으로 해결함
- 라이다 방향이 실제 로봇 방향과 맞지 않아 scan mirror와 TF yaw를 조정함
- 라이다가 멈춰 보이던 문제 중 일부는 5V/VIN 접촉 불량이었다

## IMU Bringup

ESP32에서 MPU6050 값을 UDP로 보내고 ROS2에서 `/imu/data_raw`로 변환했다.

확인한 점:

- IMU는 약 20Hz로 들어옴
- RViz와 odom 계산에 사용할 수 있음
- 처음에는 IMU z축 기준으로 기울기를 계산했더니 장착 방향 때문에 `TILT_CRITICAL` 오탐이 발생함
- 이후 부팅 직후 현재 중력 방향을 기준 자세로 저장하는 방식으로 개선함

## TF 설정

로봇 기준 프레임은 `base_link`로 잡았다.

설정한 대략적인 위치:

```text
base_link -> laser
x = 0.18m
z = 0.16m

base_link -> imu_link
x = 0.07m
z = 0.08m
```

RViz에서 라이다 점과 실제 벽 위치가 맞는지 확인하면서 라이다 방향을 조정했다.

## 주행 Bringup

ROS2 주행 명령 흐름:

```text
/cmd_vel_raw
-> lidar_safety_stop
-> /cmd_vel
-> ackermann_udp_bridge
-> ESP32 UDP 12347
-> 모터/서보 제어
```

구현/조정한 내용:

- Ackermann 조향 변환
- 모터 명령 UDP 송신
- 명령 끊김 시 자동 정지
- `/cmd_vel_executed` 발행
- 오른쪽 뒷모터가 뻑뻑해서 오른쪽 모터 보정값 추가
- 저속에서 모터 힘이 부족해서 최소 PWM 값을 올림

알게 된 점:

- 오른쪽 뒷모터가 손으로 밀 때 잘 안 도는 물리적 저항이 있었다
- 소프트웨어 보정으로 어느 정도 주행은 가능하지만, 완전한 직진성은 하드웨어 상태의 영향을 받는다

## 전원과 안정성

발견한 문제:

- 라이다가 가끔 멈추거나 오래된 점만 보임
- ESP32가 VIN 전원에서 가끔 명령을 못 받다가 USB 연결 후 다시 살아남
- 카메라도 HTTP 상태 확인 방식은 스트리밍 중 timeout이 날 수 있었음

판단:

- 라이다 문제는 5V/VIN 접촉 불량 가능성이 큼
- ESP32 문제는 전원 안정성, reset 상태, 순간 전압강하 가능성이 있음

추천 개선:

- 점퍼선 대신 단단한 커넥터 사용
- ESP32, 라이다 근처에 커패시터 추가
- 모든 장치 GND 공통 연결 확인
- reset 버튼 접근 가능하게 유지
- 전원선 흔들림 방지

## 카메라

Seeed XIAO ESP32-S3 Sense 카메라를 추가했다.

스트림 주소:

```text
http://192.168.0.11/stream
```

처음에는 ROS2에서 HTTP `/status`를 주기적으로 확인하려 했지만, 스트리밍 중 timeout이 발생했다.

해결:

- 카메라 펌웨어에서 UDP heartbeat를 추가
- ROS2 `esp32_camera_monitor`가 UDP heartbeat를 받아 `/camera/online`, `/camera/status` 발행

확인 결과:

```text
/camera/online = true
/camera/status = UDP_OK
```

## 위험 감지

`hazard_monitor`를 추가했다.

입력:

```text
/scan_viz
/imu/data_raw
```

출력:

```text
/camping_robot/hazard
/warning_buzzer
```

감지 상태:

```text
OK
SCAN_STALE
IMU_STALE
OBSTACLE_CLOSE
OBSTACLE_CRITICAL
TILT_WARN
TILT_CRITICAL
```

개선한 점:

- IMU 장착 방향에 상관없이 기울기 감지 가능하게 기준 자세 캘리브레이션 추가
- 순찰 중 몸통 박치기를 줄이기 위해 장애물 감지 거리를 늘림
- 가까운 장애물부터 부저가 울리도록 변경

현재 주요 기준:

```text
OBSTACLE_CLOSE: 0.45m
OBSTACLE_CRITICAL: 0.22m
```

## 부저

3핀 MM-FMD low-level-trigger 부저를 사용했다.

배선:

```text
VCC -> ESP32 3V3
GND -> ESP32 GND
I/O -> ESP32 GPIO13
```

처음에는 딸깍 소리만 났다.

원인 추정:

- 단순 ON/OFF 부저가 아니라 주파수를 넣어야 소리나는 passive 계열일 가능성

해결:

- `ackerman_esp32_imu_v3.ino`에서 `GPIO13`에 2400Hz tone 출력
- ROS2 `/warning_buzzer`를 ESP32 UDP `BUZZER,1`, `BUZZER,0` 명령으로 변환

테스트 결과:

- `alert` 명령으로 부저 정상 동작 확인

## 미션 상태 통합

`mission_supervisor`를 추가했다.

통합한 상태:

- 카메라 상태
- ESP32 상태
- 위험 감지 상태
- goal 상태
- patrol 상태

출력:

```text
/mission/status
/mission/level
```

상태 레벨:

```text
OK
WARN
DANGER
```

수정한 점:

- ESP32 재부팅 감지는 필요하지만, 한 번 발생했다고 영원히 순찰을 막으면 불편함
- `ESP32_REBOOT_SEEN`은 30초 후 자동 해제되도록 수정

## 미션 명령

`mission_command_node`를 추가했다.

입력:

```text
/mission/command
```

지원 명령:

```text
alert
stop
patrol
next
reset_patrol
delivery
guide
evacuate
```

동작:

- `alert`: 정지 + 부저 ON
- `stop`: 정지 + 순찰 일시정지 + 부저 OFF
- `patrol`: 미션 상태가 OK일 때 순찰 시작
- `next`: 현재 웨이포인트 건너뛰기
- `reset_patrol`: 첫 웨이포인트로 초기화
- `delivery`, `guide`, `evacuate`: 명령은 받지만 아직 실제 동작은 미구현

## 웨이포인트 순찰

웨이포인트 파일:

```text
software/ros2_ws/src/camping_robot_bringup/config/patrol_waypoints.yaml
```

RViz에서 `2D Goal Pose`로 찍은 좌표를 저장하는 `waypoint_recorder`를 추가했다.

순찰 명령:

```bash
ros2 topic pub /mission/command std_msgs/msg/String "{data: patrol}" --once
```

정지:

```bash
ros2 topic pub /mission/command std_msgs/msg/String "{data: stop}" --once
```

다음 지점으로 넘기기:

```bash
ros2 topic pub /mission/command std_msgs/msg/String "{data: next}" --once
```

## Simple Patrol의 한계

처음 순찰은 `simple_goal_follower` 기반이었다.

이 방식은 목표 방향으로 직접 가는 방식이라 맵에 이미 있는 장애물도 우회하지 못했다.

즉:

```text
맵에 있는 벽/물체를 보고 경로를 돌아가는 기능은 없음
라이다로 가까운 장애물을 보면 멈추거나 삐삐 울림
하지만 우회 경로를 새로 만들지는 못함
```

그래서 순찰 중 장애물에 몸통 박치기하는 문제가 있었다.

## Nav2 Patrol로 전환

순찰이 실제로 우회하려면 Nav2를 쓰는 것이 맞다고 판단했다.

추가한 launch:

```bash
ros2 launch camping_robot_bringup camping_robot_nav2_patrol.launch.py
```

Nav2 순찰은 다음을 사용한다.

```text
저장된 map
AMCL
global costmap
local costmap
planner
Regulated Pure Pursuit controller
NavigateToPose action
```

기대 동작:

- 맵에 이미 있는 장애물은 global planner가 우회 경로 생성
- 새로 생긴 장애물은 local/global costmap에 반영
- 장애물이 길을 막으면 정지하거나 재계획

Ackermann 조향 특성을 고려해 DWB 대신 `RegulatedPurePursuit` 컨트롤러를 적용했다.

## 오늘 배운 점

- ROS2에서 `source /opt/ros/humble/setup.bash`는 ROS2 기본 명령과 패키지를 터미널에 알려주는 과정이다.
- `source install/local_setup.bash`는 내가 빌드한 워크스페이스 패키지를 현재 터미널에 알려주는 과정이다.
- `--no-daemon` 또는 `ROS_NO_DAEMON=1`은 ROS2 daemon 때문에 topic list/echo가 멈출 때 도움이 된다.
- RViz에서 Map은 `Transient Local` QoS가 필요할 수 있다.
- 라이다/IMU/카메라가 ROS2에 보이는 것과 실제 주행 안정성은 별도 문제다.
- Ackermann 로봇은 제자리 회전이 어려워서 목표 방향 맞추기와 경로 추종이 차동구동보다 어렵다.
- 단순 goal follower는 장애물 회피가 아니다.
- 실제 순찰 로봇에는 map, costmap, planner, local controller가 필요하다.
- 전원과 커넥터 안정성은 소프트웨어만큼 중요하다.

# TIL 2026-06-12 - Camping Bot Bringup

## Goal

캠핑장을 순찰하며 위험을 감지하고, 배송/길안내/대피/경고 같은 미션으로 확장할 수 있는 ROS2 기반 캠핑 로봇의 1차 bringup을 진행했다.

## Sensor Bringup

- LDS14 lidar UDP packets were received and converted to `/scan`, `/scan_fast`, and `/scan_viz`.
- IMU packets were received on `/imu/data_raw`.
- RViz QoS issues were handled by using best-effort scan topics and transient-local map display settings.
- Lidar frame direction was corrected using static TF yaw and scan mirroring.
- Lidar and IMU static TFs were set from `base_link`:
  - lidar: `x=0.18`, `z=0.16`
  - IMU: `x=0.07`, `z=0.08`

## Drive Bringup

- ESP32 receives Ackermann drive commands over UDP.
- `/cmd_vel_raw` goes through `lidar_safety_stop`.
- Filtered `/cmd_vel` goes to `ackermann_udp_bridge`.
- `ackermann_udp_bridge` sends UDP commands to the ESP32.
- Motor minimum PWM was raised to overcome weak low-speed torque.
- Right motor trim was added because the right rear motor had higher drag.
- Command timeout was added so the robot stops if commands stop arriving.

## Power And Stability Findings

- Lidar dropouts were often caused by unstable 5V/VIN wiring or connector contact.
- ESP32 motor board sometimes needed a manual reset or USB reconnection; likely power/reset stability related.
- Recommendation:
  - use firmer connectors
  - common ground
  - add capacitors near ESP32/lidar power input
  - keep manual reset accessible

## Camera

- Seeed XIAO ESP32-S3 Sense camera was added.
- HTTP stream worked at:

```text
http://192.168.0.11/stream
```

- HTTP polling could time out while streaming, so UDP heartbeat was added.
- `/camera/online` and `/camera/status` now report camera status reliably.

## Hazard Monitor

`hazard_monitor` publishes:

- `/camping_robot/hazard`
- `/warning_buzzer`

Important fixes:

- IMU tilt detection no longer assumes gravity is on IMU z-axis.
- Startup IMU baseline calibration was added.
- The robot should stay still and level for about 2 seconds after launch.
- Obstacle thresholds were tuned for patrol:
  - close obstacle: `0.45m`
  - critical obstacle: `0.22m`
- Buzzer turns on for close and critical obstacles.

## Buzzer

- MM-FMD style 3-pin low-level-trigger buzzer was connected:

```text
VCC -> ESP32 3V3
GND -> ESP32 GND
I/O -> ESP32 GPIO13
```

- `ackerman_esp32_imu_v3.ino` outputs a `2400Hz` tone so passive-style modules can sound.
- ROS topic `/warning_buzzer` is bridged to ESP32 UDP command:

```text
BUZZER,1
BUZZER,0
```

## Mission Command

`mission_command_node` was added.

Input:

```bash
/mission/command
```

Supported commands:

- `alert`: stop robot and turn buzzer on
- `stop`: stop robot, pause patrol, turn buzzer off
- `patrol`: start waypoint patrol when mission level is OK
- `next`: skip current waypoint
- `reset_patrol`: reset patrol to the first waypoint
- `delivery`, `guide`, `evacuate`: accepted but marked as not ready

Status:

```bash
/mission/mode_status
```

## Nav2 Patrol

The key design correction:

- Simple patrol does not use the saved map for path planning.
- Therefore it can drive toward objects already visible in the map.
- Real patrol behavior should use Nav2 costmaps and a planner.

Added:

```bash
ros2 launch camping_robot_bringup camping_robot_nav2_patrol.launch.py
```

Nav2 patrol uses:

- saved map
- AMCL
- global costmap
- local costmap
- planner
- Regulated Pure Pursuit controller
- waypoint patrol in `navigation_mode=nav2`

This is the correct direction for routing around mapped obstacles and reacting to newly detected lidar obstacles.
